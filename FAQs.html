Accessibility Testing FAQ

Answers to common questions about accessibility testing services and my work

What accessibility testing services do you offer?

I provide comprehensive accessibility testing services including:

  • Web Accessibility Testing: Complete WCAG 2.0, 2.1, 2.2 compliance testing for websites
  • Mobile Accessibility Testing: iOS and Android native app testing with VoiceOver and TalkBack
  • Automated Testing: Using Axe DevTools, Lighthouse, WAVE, ARC Toolkit, and Level Access AMP
  • Manual Testing: Keyboard navigation, screen reader testing (NVDA, JAWS, VoiceOver, TalkBack)
  • VPAT/ACR Creation: Voluntary Product Accessibility Templates and Accessibility Conformance Reports
  • Accessibility Audits: Comprehensive audits with detailed remediation guidance
  • Design Review: Figma accessibility annotations and acceptance criteria
  • Training & Consultation: Team training on accessibility best practices
What is your accessibility testing process?

My testing process follows a comprehensive 4-phase methodology:

  • Phase 1 - Automated Scanning: Initial assessment using industry-standard tools (Axe, Lighthouse, WAVE) to identify common WCAG violations
  • Phase 2 - Manual Testing: Comprehensive keyboard navigation and screen reader testing across NVDA, JAWS, VoiceOver, and TalkBack to catch issues automated tools miss
  • Phase 3 - Documentation: Detailed VPAT/ACR reports with prioritized recommendations, WCAG success criteria mapping, and clear remediation guidance
  • Phase 4 - Collaboration: Working with design and development teams to implement solutions, validate fixes, and ensure compliance throughout the SDLC

This ensures thorough coverage of all WCAG 2.2 Level A/AA success criteria.

ARIA Live Regions

Interactive demo for NVDA screen reader announcements  ·  aria-live aria-atomic role="log" role="marquee"

aria-live="polite"

polite

NVDA waits until the user finishes what they're reading, then announces the change. Best for non-urgent updates like form validation, status messages.

<!-- HTML --> <div aria-live="polite" aria-atomic="false"> Dynamic content goes here </div> /* JS: just update the DOM */ document.getElementById('polite-region') .textContent = 'New message!';

aria-live="assertive"

assertive

NVDA interrupts immediately — even if the user is mid-sentence in another element. Use only for critical errors or alerts. Overuse is very disruptive.

<div aria-live="assertive" aria-atomic="true" role="alert"> </div> /* Interrupts the user immediately */ document.getElementById('assertive-region') .textContent = 'ERROR: Session expired!';

role="log"

log polite implicit

Designed for sequential, append-only content like chat logs, console output, activity feeds. Implicitly sets aria-live="polite" and aria-atomic="false". NVDA announces only newly added items, not the whole list.

<ul role="log" aria-label="Chat messages"> <!-- role="log" implicitly sets: aria-live="polite" aria-atomic="false" aria-relevant="additions text" --> <li>Alice: Hello!</li> <li>Bob: Hi there</li> </ul>

role="marquee"

marquee off implicit

For continuously cycling, non-critical content like tickers, ads, carousels. Implicitly sets aria-live="off" — NVDA does NOT auto-announce it. Users can navigate to it manually. Use sparingly; not ideal for important info.

<div role="marquee" aria-label="News ticker"> <!-- role="marquee" implicitly sets: aria-live="off" (NVDA does NOT auto-read this!) User must navigate here manually --> Breaking: Market up 2% today </div>

aria-atomic="true" vs "false"

atomic

atomic=true: NVDA reads the entire region whenever any part changes — good for clocks, status bars, composite widgets.

atomic=false (default): NVDA reads only the changed node — good for chat logs, lists.

<!-- atomic=true: reads WHOLE region --> <div aria-live="polite" aria-atomic="true"> Score: <span id="score">0</span> / 10 </div> <!-- NVDA says: "Score: 5 / 10" (full text) --> <!-- atomic=false: reads ONLY changed part --> <div aria-live="polite" aria-atomic="false"> Score: <span id="score2">0</span> </div> <!-- NVDA says: "5" (only the span) -->

aria-live="off"

off silent

The default for all elements. NVDA does not announce changes automatically. Content is readable only when the user navigates to it. This is the correct setting for decorative or non-critical dynamic content.

<div aria-live="off"> <!-- Default for ALL elements. NVDA only reads this when the user navigates here. No automatic announcements. --> Decorative / non-critical content </div>

📋 Quick Reference — NVDA Behavior Summary

Role / Attribute Implicit aria-live Implicit aria-atomic NVDA auto-announces? Interrupts reading? Best used for
aria-live="polite" polite false ✓ Yes (when idle) ✗ No Form validation, status, notifications
aria-live="assertive" assertive false ✓ Yes (immediately) ✓ Yes! Errors, critical alerts only
aria-live="off" off false ✗ No ✗ No Default; decorative content
role="log" polite false ✓ Yes (new entries) ✗ No Chat, activity feed, console output
role="marquee" off ⚠ false ✗ No (silent!) ✗ No Tickers, carousels (use with caution)
role="alert" assertive true ✓ Yes (immediately) ✓ Yes! Error messages, critical notices
aria-atomic="true" (depends on live) true Reads full region ✗ Clocks, scoreboards, composite status
aria-atomic="false" (depends on live) false Reads only changed nodes ✗ Lists, logs, append-only content
🎧 NVDA Testing Tips: Enable speech viewer via NVDA menu → Tools → Speech Viewer to see what NVDA would announce without headphones. Use NVDA + F5 to list all live regions on the page. Always test assertive sparingly — it interrupts the user and creates a poor experience if overused.

⚠ Common Mistake: Adding aria-live to an element after page load works. But the live region must exist in the DOM before you inject content into it — don't create the region and populate it in the same JS tick.
