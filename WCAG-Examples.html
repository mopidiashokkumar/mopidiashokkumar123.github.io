ARIA Working Examples

Live, copy-paste code with pass & fail examples and explanations for your team.

role="presentation" role="none" aria-atomic role="log" role="marquee"
1
role="presentation"
Remove an element's semantic meaning from the accessibility tree
What it does: Strips the native ARIA role from an element. Screen readers will still see the element's text content, but won't announce what type of element it is (e.g., won't say "table", "list", "separator"). Normally, <table> announces "table", <hr> announces "separator" — adding role="presentation" silences those role announcements.
⚠️
Golden rule: Only use on elements that are purely decorative and have no interactive children. If a screen reader user needs to understand or act on it — do NOT strip the role.
PASS — Layout table (decorative)
<!-- CORRECT: table used only for visual layout -->
<table role="presentation">
  <tr>
    <td>
      <img src="logo.png" alt="Company Logo">
    </td>
    <td>
      <nav>...</nav>
    </td>
  </tr>
</table>
✓ Why it passes: This table holds layout, not tabular data. Removing "table" role prevents screen readers from announcing row/column counts that would confuse users. The actual meaningful content (nav, image) still has its own semantics. 
<!-- CORRECT: decorative divider -->
<hr role="presentation">
✓ Why it passes: A visual separator between sections with no meaningful semantic purpose. Stripping "separator" stops the screen reader announcing it as a landmark.
FAIL — Real data table stripped
<!-- WRONG: this IS a real data table -->
<table role="presentation">
  <caption>Q1 Sales Results</caption>
  <thead>
    <tr>
      <th>Product</th>
      <th>Revenue</th>
    </tr>
  </thead>
  <tbody>...</tbody>
</table>
✗ Why it fails: This IS a real data table. Screen reader users lose all row/column context, can't navigate by cell, and the caption is hidden. They'll hear the cell text with zero structure — a critical accessibility failure. 
<!-- WRONG: interactive button stripped -->
<button role="presentation">
  Submit Form
</button>
✗ Why it fails: Stripping the role from an interactive element removes its "button" announcement. Screen reader users won't know they can press it. Never use on focusable elements.
2
role="none"
The modern synonym for role="presentation" — use this in all new code
What it does: Exactly the same as role="presentation" — removes an element's native semantic role from the accessibility tree. The ARIA spec treats them as identical synonyms. Use role="none" in new code — it more clearly communicates intent ("this element is nothing special"). Some teams write role="none presentation" for maximum browser compatibility.
ValueMeaningWhich to use
role="presentation"Original name in ARIA 1.0Legacy code — still valid
role="none"Synonym added in ARIA 1.1All new code — preferred
role="none presentation"Both names as fallback listMaximum browser compatibility
PASS — Nav list cleaned up
<!-- CORRECT: <ul> is just CSS structure here.
     Semantics come from <nav> and <a> tags. -->
<nav aria-label="Main navigation">
  <ul role="none">
    <li><a href="/">Home</a></li>
    <li><a href="/about">About</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>
✓ Why it passes: The <nav> landmark provides context; the <a> tags provide link semantics. The <ul> is pure CSS structure. role="none" removes the redundant "list" announcement without losing anything meaningful. 
<!-- CORRECT: decorative icon, fully hidden -->
<span role="none" aria-hidden="true">
  <svg ...>...</svg>
</span>
<!-- aria-hidden removes it from tree entirely -->
✓ Why it passes: role="none" strips the span's role; aria-hidden="true" removes it from the accessibility tree completely. The decorative icon is invisible to screen readers.
FAIL — Landmark elements stripped
<!-- WRONG: nav is a landmark — never strip it -->
<nav role="none">
  <a href="/">Home</a>
  <a href="/about">About</a>
</nav>
✗ Why it fails: Screen reader users use landmarks (<nav>, <main>, <header>, <footer>) to quickly jump around a page. Removing the "navigation" role destroys this shortcut. Never strip landmark elements. 
<!-- WRONG: headings build page structure -->
<h2 role="none">Our Services</h2>
<h3 role="none">Web Design</h3>
✗ Why it fails: Screen readers use headings to build a page outline and let users jump between sections. Removing heading roles destroys the page structure entirely. Users lose the ability to navigate to any section.
3
aria-atomic
Control how much of a live region gets announced when it changes
What it does: When content inside an aria-live region changes, aria-atomic tells the screen reader whether to announce only the changed part (false, default) or the entire region as one unit (true). This is critical for compound data like prices, scores, or counters where a partial reading would lose context.
ValueWhat gets announcedBest for
aria-atomic="false"Only the changed node (default)Chat logs, activity feeds, sequential messages
aria-atomic="true"The entire region as one unitStock prices, scores, counters with units, compound data
PASS — Stock price with atomic="true"
<!-- CORRECT: whole region announced together -->
<div
  aria-live="polite"
  aria-atomic="true"
  aria-label="AAPL stock price"
>
  <span id="price">€ 171.23</span>
  <span id="delta">+1.4%</span>
</div>

<!-- When delta updates, screen reader announces:
     "AAPL stock price € 171.23 +1.4%"
     NOT just "+1.4%" -->
✓ Why it passes: When only #delta changes, hearing "+1.4%" alone is meaningless. With atomic="true", the entire group is announced: "€ 171.23 +1.4%" — full context every time.
▶ live demo — click to simulate update
€ 171.23 +1.4%
Screen reader will announce the full region when updated.
FAIL — Missing atomic on compound data
<!-- WRONG: atomic not set (defaults to false) -->
<div
  aria-live="polite"
  <!-- aria-atomic missing! -->
>
  <span id="price">€ 171.23</span>
  <span id="delta">+1.4%</span>
</div>

<!-- When delta updates, screen reader says:
     "+1.4%" — user has no context! -->
✗ Why it fails: Without aria-atomic="true", only the changed child (#delta) is announced. The user hears "+0.8%" with no idea what stock or what price it refers to. Partial context = confusion. 
<!-- WRONG: atomic="true" on a chat log -->
<ul
  role="log"
  aria-live="polite"
  aria-atomic="true"
>
  <li>Alice: Hello!</li>
  <li>Bob: Hi there</li>
  <li>Priya: Welcome :)</li>
</ul>
✗ Why it fails: Every new message triggers a full re-read of all previous messages. Users hear the entire history again each time. For logs, keep default atomic="false" so only the new entry is announced.
4
role="log"
A polite live region for sequential, additive messages
What it does: Marks a region as a sequential log (like chat, activity feed, or console output). Screen readers announce new entries politely — they wait until the user finishes reading before announcing — and announcements are one entry at a time (aria-atomic="false" implied). Implies aria-live="polite" by default.
💡
Use for: chat messages, activity feeds, deployment logs, audit trails, notification history, server console output — anything where entries accumulate and order matters.
PASS — Chat message feed
<!-- CORRECT: polite sequential log -->
<section
  role="log"
  aria-label="Team chat"
  aria-live="polite"
>
  <div class="message">
    <strong>Alice:</strong> Hello team!
  </div>
  <div class="message">
    <strong>Bob:</strong> Hey Alice!
  </div>
</section>

<!-- Adding a new .message div to the section
     announces just that message. -->
✓ Why it passes: Messages are additive, ordered, and non-urgent. role="log" + aria-live="polite" means each new message is announced when the screen reader is idle — no interruptions, no history re-reads.
▶ live demo — click to add messages
AliceHey team, build is looking good 🚀
FAIL — assertive on a chat log
<!-- WRONG: assertive interrupts the user -->
<div
  role="log"
  aria-live="assertive"
>
  <p>Alice: Hello!</p>
</div>
✗ Why it fails: assertive interrupts whatever the user is currently reading — every chat message becomes an interruption. This is disruptive for a log. Only use assertive for critical alerts like error messages. 
<!-- WRONG: role="alert" for a log -->
<div role="alert">
  <p>Alice: Morning!</p>
  <p>Bob: Hey Alice</p>
  <p>Priya: Hi all!</p>
</div>
✗ Why it fails: role="alert" is assertive AND atomic — it re-reads the entire region on every change. Every new chat message triggers the whole history. It also implies urgency that regular chat doesn't have.
5
role="marquee"
Non-essential, continuously scrolling content — not announced automatically
What it does: Marks a live region whose content is non-essential and continuously updating — like a stock ticker or news banner. Unlike role="log", this implies aria-live="off" by default — screen readers do not announce updates automatically. Users can still manually navigate into the region to read it when they choose.
💡
Key distinction: role="marquee" = non-critical scrolling noise users can ignore. role="alert" = urgent information that needs immediate attention. Never confuse the two.
PASS — Stock ticker banner
<!-- CORRECT: aria-live="off" — no auto-announce -->
<div
  role="marquee"
  aria-label="Live stock prices"
  aria-live="off"
>
  <span>AAPL €171.23 ▲</span>
  <span>GOOG €142.10 ▼</span>
  <span>MSFT €415.70 ▲</span>
</div>

<!-- Ticker updates are silent.
     User can focus/navigate in to read it.
     aria-label identifies region on focus. -->
✓ Why it passes: Tickers scroll continuously — announcing every update would be constant noise. aria-live="off" keeps it silent. aria-label identifies the region when a user manually explores it. Users can choose to read it when they want.

FAIL — Alert on ticker / no role
<!-- WRONG: alert on a scrolling ticker -->
<div
  role="alert"
  aria-live="assertive"
>
  AAPL €171.23 ▲ | GOOG €142.10 ▼
</div>
✗ Why it fails: Every tick update triggers an assertive, atomic announcement — the screen reader constantly interrupts to read stock prices. For continuously updating non-critical content this makes the page completely unusable. 
<!-- WRONG: aria-live="polite" on a ticker -->
<div
  <!-- no role="marquee" -->
  aria-live="polite"
>
  AAPL €171.23 | GOOG €142.10
</div>

<!-- No aria-label — unidentified region.
     Polite still announces each tick update. -->
✗ Why it fails: Missing role="marquee" means assistive tech doesn't understand the context. Even aria-live="polite" announces each tick update when the user pauses — still disruptive for a ticker. Use aria-live="off".
